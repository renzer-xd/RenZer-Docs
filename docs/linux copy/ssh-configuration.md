# คู่มือการตั้งค่าและใช้งาน SSH (Secure Shell) อย่างละเอียด

**SSH (Secure Shell)** เป็นโปรโตคอลเครือข่ายมาตรฐานที่ใช้สำหรับเข้ารหัสสัญญาณการเชื่อมต่อเพื่อเข้าควบคุม สั่งการ หรือบริหารจัดการเครื่องเซิร์ฟเวอร์ปลายทางระยะไกลอย่างปลอดภัย 

ในคู่มือนี้จะอธิบายขั้นตอนการใช้งานแบบเจาะลึก ตั้งแต่การสร้างกุญแจเข้ารหัส (SSH Key) บนระบบ Windows, การนำคีย์ขึ้นไปติดตั้งที่ฝั่งเซิร์ฟเวอร์ Linux, การเชื่อมต่อผ่านช่องทางต่างๆ ไปจนถึงการปรับแต่งความปลอดภัยระดับสูงสุดของ SSH Daemon (Hardening) และการย่นเวลาเชื่อมต่อด้วยไฟล์ Config

---

## 🔑 1. การสร้าง SSH Key บนระบบปฏิบัติการ Windows

การล็อกอินด้วยรหัสผ่านธรรมดาเสี่ยงต่อการถูก Brute-force สุ่มรหัสผ่านได้ตลอดเวลา **SSH Key Pair** จึงเป็นวิธีที่ปลอดภัยกว่ามาก โดยทำงานด้วยหลักการ "กุญแจคู่แบบอสมมาตร":

* **Private Key (กุญแจส่วนตัว):** เก็บไว้ที่เครื่องตัวเอง (Windows) เท่านั้น — ห้ามส่งต่อหรือแชร์ให้ใครเด็ดขาด
* **Public Key (กุญแจสาธารณะ):** นำไปติดตั้งไว้ที่เซิร์ฟเวอร์ปลายทาง — เปรียบเหมือนรูแม่กุญแจที่รอการไข เมื่อ Private Key ตรงกันจึงจะเปิดผ่านได้

### ขั้นตอนการสร้าง SSH Key ผ่าน Windows Terminal (PowerShell หรือ Command Prompt):

1. เปิดแอปพลิเคชัน **PowerShell** หรือ **Command Prompt** บนเครื่อง Windows ของคุณ
2. พิมพ์คำสั่งสร้างคีย์ (แนะนำประเภท **ED25519** ซึ่งเป็นอัลกอริทึมยุคใหม่ที่มีความปลอดภัยสูงและประมวลผลเร็วกว่า RSA):
   ```bash
   ssh-keygen -t ed25519 -C "your_email@example.com"
   ```
   *(ค่า `-C` คือ Comment ที่ติดท้ายคีย์เพื่อบอกว่าคีย์นี้เป็นของใคร เปลี่ยนเป็นอีเมลหรือชื่อตัวเองได้)*

   > [!TIP]
   > หากเซิร์ฟเวอร์ปลายทางเก่ามากและยังไม่รองรับ ED25519 ให้สร้างคีย์แบบ **RSA 4096 บิต** แทน:
   > ```bash
   > ssh-keygen -t rsa -b 4096 -C "your_email@example.com"
   > ```

3. **กำหนดตำแหน่งจัดเก็บไฟล์:**
   ระบบจะแสดงคำถามถามตำแหน่งจัดเก็บไฟล์คีย์ย่อย:
   ```text
   Enter file in which to save the key (C:\Users\username/.ssh/id_ed25519):
   ```
   *แนะนำให้กดปุ่ม **`Enter`** ข้ามไปเลย* เพื่อยอมรับการเซฟลงตำแหน่งโฟลเดอร์หลักเริ่มต้นของ Windows (ซึ่งก็คือโฟลเดอร์ `.ssh` ใต้บัญชีผู้ใช้ของคุณ)

4. **กำหนดรหัสผ่านล็อกคีย์ (Passphrase):**
   ระบบจะถามให้กรอกรหัสผ่านเพื่อล็อกไฟล์กุญแจส่วนตัวไว้อีกชั้น:
   ```text
   Enter passphrase (empty for no passphrase):
   ```
   * **กรณีต้องการความสะดวก:** กดปุ่ม **`Enter` ผ่านไปทันที** (ไม่ต้องกรอกรหัส) เพื่อให้เวลาเชื่อมต่อ SSH จะกระโดดล็อกอินเข้าเครื่องปลายทางได้ทันทีโดยไม่ต้องถามรหัสใดๆ
   * **กรณีต้องการความปลอดภัยสูงสุด:** กรอกรหัสผ่านที่คุณต้องการล็อกไฟล์คีย์เอาไว้ (เวลาใช้งานระบบจะบังคับถามรหัสผ่านนี้ก่อนเปิดใช้คีย์ไข)

5. **ตรวจสอบไฟล์ที่สร้างขึ้น:**
   ให้เปิดโปรแกรม File Explorer ไปที่ตำแหน่งพาธ `C:\Users\<ชื่อผู้ใช้งานของคุณ>\.ssh\` จะพบไฟล์คู่หูเกิดขึ้นมา 2 ไฟล์:
   * **`id_ed25519`** : ไฟล์กุญแจส่วนตัว (Private Key)
   * **`id_ed25519.pub`** : ไฟล์กุญแจสาธารณะ (Public Key)

## 📤 2. การติดตั้ง Public Key ไปยังฝั่งเซิร์ฟเวอร์ Linux (Manual Installation)

เมื่อคุณได้คีย์บน Windows แล้ว ขั้นตอนถัดไปคือการนำข้อความในไฟล์กุญแจสาธารณะ `.pub` ไปบันทึกเก็บไว้ที่บัญชีผู้ใช้บนเซิร์ฟเวอร์ปลายทาง

เนื่องจากฝั่ง Windows ไม่มีคำสั่งติดตั้งอัตโนมัติอย่าง `ssh-copy-id` เราจึงสามารถติดตั้งด้วยตนเอง (Manual) ได้ตามขั้นตอนดังนี้:

1. **เปิดอ่านข้อความใน Public Key:**
   ในหน้าต่าง PowerShell รันคำสั่งนี้เพื่อดึงเนื้อหาข้อความทั้งหมดมาแสดงหน้าจอ:
   ```powershell
   Get-Content C:\Users\<ชื่อผู้ใช้งานของคุณ>\.ssh\id_ed25519.pub
   ```
   *(หรือใช้โปรแกรม Notepad บน Windows เปิดไฟล์ `id_ed25519.pub` แล้วคัดลอกข้อความยาวๆ ทั้งบรรทัดเตรียมไว้)*

2. **ล็อกอินเข้าเซิร์ฟเวอร์ Linux และสร้างโฟลเดอร์เก็บคีย์:**
   ล็อกอินเข้าเซิร์ฟเวอร์ด้วยรหัสผ่านปกติก่อน จากนั้นไปที่ Home Directory ของผู้ใช้ที่ต้องการติดตั้ง แล้วรันคำสั่ง:
   ```bash
   # สร้างโฟลเดอร์ .ssh (หากยังไม่มีในเครื่อง)
   mkdir -p ~/.ssh
   
   # กำหนดสิทธิ์ให้เฉพาะเจ้าของบัญชีเท่านั้นที่เปิดดูและจัดการสิทธิ์ของโฟลเดอร์นี้ได้ (สิทธิ์ 700)
   chmod 700 ~/.ssh
   ```

3. **เขียนคีย์ลงไฟล์ `authorized_keys`:**
   เปิดไฟล์สำหรับเก็บกุญแจที่ได้รับอนุญาตด้วยคำสั่ง:
   ```bash
   nano ~/.ssh/authorized_keys
   ```
   จากนั้นทำการ **กดวาง (Paste) ข้อความคีย์ยาวๆ** ที่คัดลอกมาจากไฟล์ `id_ed25519.pub` ลงไปในไฟล์นี้ (หากมีคีย์ของอุปกรณ์เครื่องอื่นอยู่ก่อนแล้ว ให้กด Enter เพื่อขึ้นบรรทัดใหม่แล้ววางต่อท้ายได้เลย)
   
   เสร็จแล้วบันทึกไฟล์และกดปิดออก (`Ctrl + O` -> `Enter` -> `Ctrl + X`)

4. **ปรับสิทธิ์ความปลอดภัยให้แก่ไฟล์:**
   เพื่อให้ระบบ SSH ยอมรับคีย์นี้ ไฟล์เก็บคีย์ต้องได้รับการกำหนดสิทธิ์ที่ปลอดภัยห้ามมิให้ผู้อื่นแก้ไขอ่านได้เด็ดขาด:
   ```bash
   chmod 600 ~/.ssh/authorized_keys
   ```

---

> [!TIP]
> **ทางลัดสำหรับผู้ใช้ Linux/Mac:** หากเครื่องต้นทางเป็น Linux หรือ macOS มีคำสั่ง `ssh-copy-id` ที่ทำขั้นตอนที่ 2-4 ข้างต้นให้อัตโนมัติในคำสั่งเดียว:
> ```bash
> # ติดตั้ง Public Key ไปยังเซิร์ฟเวอร์แบบอัตโนมัติ
> ssh-copy-id -i ~/.ssh/id_ed25519.pub username@server_ip
>
> # กรณีเซิร์ฟเวอร์ใช้พอร์ตที่กำหนดเอง
> ssh-copy-id -i ~/.ssh/id_ed25519.pub -p 22022 username@server_ip
> ```
> *(คำสั่งนี้จะสร้างโฟลเดอร์ `.ssh`, เขียนคีย์ลง `authorized_keys`, และตั้งสิทธิ์ที่ถูกต้องให้ครบในทีเดียว)*

หลังจากที่ผูกคีย์สาธารณะ (Public Key) ไว้บนฝั่ง Linux เรียบร้อยแล้ว คุณสามารถเลือกใช้วิธีล็อกอินเข้าเซิร์ฟเวอร์จากเครื่อง Windows ได้ผ่าน 2 ช่องทางหลักดังนี้:

---

### 🟢 ช่องทางที่ 1: การเข้าใช้งานผ่าน Command Prompt (CMD) หรือ PowerShell (Native OpenSSH)

Windows 10 และ 11 จะมีโปรแกรม OpenSSH Client ติดตั้งมาพร้อมตัวเครื่องแต่แรกอยู่แล้ว ทำให้สามารถพิมพ์คำสั่งเชื่อมต่อโดยตรงผ่าน CMD หรือ PowerShell ได้ทันที:

* **1. การเชื่อมต่อปกติ (กรณีใช้คีย์เริ่มต้นในโฟลเดอร์ `.ssh`):**
  ```bash
  ssh username@server_ip
  ```
  *(ระบบจะดึงไฟล์คีย์ส่วนตัวใน `C:\Users\username\.ssh\` มายืนยันตัวตนให้อัตโนมัติ หากสำเร็จจะกระโดดเข้าหน้าเซิร์ฟเวอร์โดยไม่ต้องกรอกรหัสผ่านใดๆ)*

* **2. การเชื่อมต่อกรณีเปลี่ยนพอร์ตบริการใหม่ (Custom Port):**
  หากคุณปรับเปลี่ยนพอร์ตของเซิร์ฟเวอร์เพื่อความปลอดภัย (เช่น พอร์ต 22022) ให้ใช้ออปชัน **`-p` (ตัวเล็ก)** ระบุหมายเลขพอร์ต:
  ```bash
  ssh -p 22022 username@server_ip
  ```

* **3. การเชื่อมต่อโดยระบุไฟล์กุญแจ (Custom Private Key Path):**
  หากคุณเซฟไฟล์กุญแจไว้ตำแหน่งอื่นที่ไม่ใช่โฟลเดอร์หลัก หรือเปลี่ยนชื่อคีย์ ให้ใช้ตัวเลือก **`-i`** ระบุที่อยู่ไฟล์กุญแจส่วนตัว:
  ```bash
  ssh -i "C:\path\to\your_key" username@server_ip
  ```

* **4. การป้อนชื่อสิทธิ์เข้าใช้งานระดับ Root โดยตรง:**
  ```bash
  ssh root@server_ip
  # หรือหากระบุพอร์ต
  ssh -p 22022 root@server_ip
  ```

---

### 🔵 ช่องทางที่ 2: การเข้าใช้งานผ่านโปรแกรม PuTTY

**PuTTY** เป็นโปรแกรมจัดการ SSH ยอดนิยมบน Windows แต่มีข้อกำหนดเฉพาะคือ **PuTTY ไม่รองรับการดึงไฟล์กุญแจส่วนตัวแบบ OpenSSH มาตรฐานโดยตรง** (เช่น ไฟล์ `id_ed25519` หรือ `id_rsa`) แต่จะทำงานร่วมกับไฟล์สกุล **`.ppk` (PuTTY Private Key)** เท่านั้น ดังนั้นเราจึงต้องแปลงรูปแบบคีย์ก่อนเริ่มการเชื่อมต่อ:

#### 🛠️ ขั้นตอนที่ 1: แปลงไฟล์กุญแจ OpenSSH เป็นคีย์ `.ppk` ด้วย PuTTYgen
1. เปิดแอปพลิเคชัน **PuTTYgen (PuTTY Key Generator)** ซึ่งจะติดมาพร้อมตอนที่คุณดาวน์โหลดติดตั้ง PuTTY
2. คลิกปุ่ม **`Load`** บริเวณกลางหน้าต่างโปรแกรม
3. จะมีหน้าต่างให้เลือกไฟล์เปิด:
   - ปรับเปลี่ยนฟิลเตอร์ประเภทไฟล์บริเวณมุมขวาล่างจากเดิมที่เป็น `PuTTY Private Key files (*.ppk)` ให้สลับเป็น **`All Files (*.*)`**
   - ไปยังตำแหน่งโฟลเดอร์เก็บคีย์ของคุณ เช่น `C:\Users\<ชื่อผู้ใช้งาน>\.ssh\`
   - เลือกไฟล์กุญแจส่วนตัวของคุณ (เช่น **`id_ed25519`** หรือ **`id_rsa`** *ระวังห้ามเลือกไฟล์ที่มีนามสกุล .pub*) แล้วกด Open
4. ป้อนรหัสผ่านปลดล็อก (Passphrase) ในกรณีที่คุณเคยตั้งค่ารหัสผ่านล็อกคีย์ไว้ตอนสร้างคีย์
5. ระบบจะขึ้นข้อความแจ้งเตือนสีเขียวว่าโหลดคีย์สำเร็จ ให้กด **OK**
6. คลิกปุ่ม **`Save private key`**
7. โปรแกรมจะแจ้งถามเพื่อความแน่ใจในกรณีที่คุณไม่ได้ตั้ง Passphrase ล็อกคีย์ไว้ ให้กด **Yes** ยืนยัน
8. ตั้งชื่อและบันทึกไฟล์เก็บไว้ในโฟลเดอร์เดิม เช่น ตั้งชื่อไฟล์เป็น **`id_ed25519.ppk`** แล้วกดบันทึก

#### 📡 ขั้นตอนที่ 2: ตั้งค่าเชื่อมต่อในโปรแกรม PuTTY เพื่อเข้าใช้คีย์
1. เปิดโปรแกรม **PuTTY**
2. ไปที่เมนู **Session** (หัวข้อแรกสุดฝั่งซ้ายมือ):
   - **Host Name (or IP address):** พิมพ์ IP ของเซิร์ฟเวอร์ (ตัวอย่าง: `103.224.86.15` หรือระบุผู้ใช้ไว้ด้านหน้าแบบ `root@103.224.86.15`)
   - **Port:** ป้อนพอร์ตเซิร์ฟเวอร์ (เช่น `22` หรือ `22022` ตามที่ปรับแต่งไว้)
   - **Connection type:** เลือกเป็น `SSH`
3. ไปที่เมนูนำทางด้านซ้ายเพื่อเรียกใช้คีย์ `.ppk`:
   - ในกล่อง Category เลื่อนลงมาเปิดหัวข้อ **Connection** -> **SSH** -> **Auth** -> **Credentials**
   - *(หมายเหตุ: หากเป็นโปรแกรม PuTTY เวอร์ชันเก่า เมนูจะอยู่ภายใต้ **Connection** -> **SSH** -> **Auth** โดยตรง)*
   - ในช่อง **Private key file for authentication:** คลิกปุ่ม **`Browse...`**
   - เลือกไฟล์ **`.ppk`** ที่คุณทำการแปลงคีย์เก็บไว้เรียบร้อยแล้วในขั้นตอนแรก (เช่น `id_ed25519.ppk`) แล้วกดเปิด
4. ทำการเซฟการตั้งค่าไว้ใช้งานรอบหน้า (ไม่ต้องตั้งค่าซ้ำ):
   - คลิกย้อนกลับมาที่เมนูแรกสุด **Session** (ฝั่งซ้ายบนสุด)
   - ในช่อง **Saved Sessions:** ตั้งชื่อเซิร์ฟเวอร์ของคุณ (เช่น `My Client Server`)
   - คลิกปุ่ม **`Save`** ด้านขวามือ
5. เริ่มเชื่อมต่อโดยกดปุ่ม **`Open`** ด้านล่างสุด
6. เมื่อหน้าต่าง Terminal ดำๆ เปิดขึ้นมาในครั้งแรก ระบบจะแสดงหน้าต่างแจ้งเตือนความปลอดภัยลายพิมพ์นิ้วมือโฮสต์ (Host Key Warning) ให้กด **Accept** หรือ **Yes** ยืนยัน เพื่อเข้าสู่หน้าจอเซิร์ฟเวอร์ Linux และใช้งานคำสั่งได้ทันที

---

## 🔒 4. การปรับแต่งเพิ่มความปลอดภัยขั้นสูงสุดให้กับบริการ SSH Daemon

เพื่อเพิ่มระดับความมั่นคงปลอดภัย ป้องกันการสแกนหาพอร์ต และปิดช่องโหว่การโจมตี แนะนำให้เปิดเข้าไฟล์คอนฟิกหลักของ SSH บนเซิร์ฟเวอร์และตั้งค่าการทำงานดังนี้:

1. เปิดเข้าไฟล์ตั้งค่าด้วยสิทธิ์สูงสุด:
   ```bash
   sudo nano /etc/ssh/sshd_config
   ```

2. **ปรับแต่งหัวข้อตั้งค่าสำคัญ (แก้ไขตามพารามิเตอร์ด้านล่าง):**

   * **เปลี่ยนพอร์ตบริการเพื่อหนีบอทสแกน (Change Port):**
     ค้นหาบรรทัด `#Port 22` -> ลบเครื่องหมาย `#` ออก แล้วเปลี่ยนหมายเลขพอร์ตเป็นพอร์ตอื่นที่คุณกำหนดเอง (พอร์ตระหว่าง 1024 ถึง 65535 เช่น 22022) เพื่อหลีกเลี่ยงบอทสแกนพอร์ตมาตรฐาน 22:
     ```text
     Port 22022
     ```

   * **ปิดการล็อกอินด้วยรหัสผ่าน บังคับใช้เฉพาะ SSH Key (Disable Password Login):**
     ค้นหาบรรทัด `PasswordAuthentication` แก้เป็น `no` เพื่อบังคับให้ผู้รีโมทเข้าได้เฉพาะเครื่องที่ลงทะเบียนคีย์คู่เอาไว้เท่านั้น ใครไม่มีคีย์จะหมดสิทธิ์เข้าถึง:
     ```text
     PasswordAuthentication no
     ```

   * **การจัดการสิทธิ์การเข้าถึงของ Root (Root Login Restrictions):**
     หากคุณสร้างผู้ใช้งานทั่วไปที่มีสิทธิ์ `sudo` แล้ว แนะนำให้ล็อกการล็อกอินโดยตรงด้วยบัญชี root เพื่อป้องกันการคาดเดาชื่อผู้ใช้:
     ```text
     PermitRootLogin no
     ```
     *(หรือหากยังจำเป็นต้องรันเป็น root ให้เปิดเฉพาะล็อกอินผ่าน SSH Key เท่านั้นโดยใช้คำสั่ง `PermitRootLogin prohibit-password`)*

3. **รีสตาร์ตเพื่อใช้งานค่าคอนฟิกใหม่:**
   ```bash
   sudo systemctl restart sshd
   ```
   > [!WARNING]
   > **ข้อปฏิบัติที่ห้ามละเลยเด็ดขาด (Safety Rule):**
   > หลังจากกดรีสตาร์ตบริการ SSH ไปแล้ว **ห้ามทำการปิดหน้าต่างเซสชัน SSH เดิมที่คุณกำลังเชื่อมต่ออยู่ในขณะนั้นโดยทันที!** 
   > ให้ทำการเปิดหน้าต่าง Terminal หรือ PowerShell บน Windows บานใหม่ขึ้นมาอีกอัน แล้วลองรันคำสั่งเชื่อมต่อ SSH ด้วยพอร์ตหรือคีย์อันใหม่ที่เพิ่งปรับแก้ไป เพื่อยืนยันว่าสามารถเข้าระบบได้จริง หากเกิดเหตุขัดข้องล็อกอินไม่ได้ คุณจะยังสามารถแก้ไขกลับมาผ่านหน้าต่างเซสชันเดิมที่ยังไม่ถูกตัดขาดการเชื่อมต่อได้ทันท่วงที

## 🚀 5. การสร้างทางลัดช่วยเชื่อมต่อ (SSH Client Config) บน Windows

หากคุณมีเซิร์ฟเวอร์หลายเครื่อง หรือขี้เกียจจำหมายเลข IP พอร์ต และต้องพิมพ์ระบุเส้นทางไฟล์คีย์ยาวเหยียดทุกครั้ง คุณสามารถสร้างระบบทางลัด (Alias) บน Windows ได้ง่ายๆ ดังนี้:

1. บนเครื่อง Windows ให้เปิดเข้าไปสร้างไฟล์ชื่อ **`config`** (ไม่มีนามสกุล) ภายใต้โฟลเดอร์ `C:\Users\<ชื่อผู้ใช้ของคุณ>\.ssh\` ด้วยโปรแกรม Notepad
2. เขียนกฎจัดรูปแบบการเข้าถึงเซิร์ฟเวอร์ของคุณดังนี้:
   ```text
   # เครื่องเซิร์ฟเวอร์ที่ 1 (เซิร์ฟเวอร์หลักของลูกค้า)
   Host clientserver
       HostName 103.224.86.15
       User root
       Port 22022
       IdentityFile ~/.ssh/id_ed25519

   # เครื่องเซิร์ฟเวอร์ที่ 2 (ทดสอบระบบ)
   Host testserver
       HostName 192.168.1.50
       User server
       Port 22
       IdentityFile ~/.ssh/id_rsa
   ```
3. บันทึกและปิดไฟล์
4. หลังจากนี้ล็อกอินเข้าเซิร์ฟเวอร์ได้ด้วยชื่อสั้นๆ คำสั่งเดียว:
   ```bash
   ssh clientserver
   ```
   *(ระบบจะดึง IP, Port, User และ Private Key จากไฟล์ config ให้อัตโนมัติ)*

## 🗂️ 6. การใช้ SSH Key หลายคู่กับหลายเซิร์ฟเวอร์

โดย default เครื่องคุณจะมีคีย์คู่เดียว (`id_ed25519`) ซึ่งใช้ได้กับทุกเซิร์ฟเวอร์ แต่ในงานจริงที่มีเซิร์ฟเวอร์หลายเครื่องหรือหลายองค์กร การสร้าง **คีย์แยกต่างหากสำหรับแต่ละเซิร์ฟเวอร์** เป็นแนวปฏิบัติที่ดีกว่า เพราะ:

* ถ้าคีย์หนึ่งรั่วไหล เซิร์ฟเวอร์อื่นยังปลอดภัย — ถอนเฉพาะคีย์นั้นออกได้ทันที
* บอกได้ชัดเจนว่าคีย์ไหนใช้กับที่ไหน ตรวจสอบง่าย
* จัดการวันหมดอายุและการ Rotate Key ได้เป็นรายเซิร์ฟเวอร์

---

### A. การสร้าง SSH Key หลายคู่พร้อมตั้งชื่อไฟล์

เวลาสร้างคีย์ ให้ **ตั้งชื่อไฟล์ให้สื่อความหมาย** แทนที่จะใช้ชื่อ default (`id_ed25519`) เหมือนกันทุกคู่:

```bash
# สร้างคีย์สำหรับ Production Server
ssh-keygen -t ed25519 -C "prod-server" -f ~/.ssh/id_ed25519_prod

# สร้างคีย์สำหรับ Staging Server
ssh-keygen -t ed25519 -C "staging-server" -f ~/.ssh/id_ed25519_staging

# สร้างคีย์สำหรับ GitHub
ssh-keygen -t ed25519 -C "your_email@example.com" -f ~/.ssh/id_ed25519_github
```

*(ตัวเลือก `-f` คือการระบุชื่อและตำแหน่งไฟล์คีย์โดยตรง โดยไม่ต้องรอให้ระบบถาม)*

หลังจากสร้างเสร็จ โฟลเดอร์ `.ssh` จะมีคีย์แยกกันชัดเจน:
```text
C:\Users\username\.ssh\
├── id_ed25519_prod         ← Private Key สำหรับ Production
├── id_ed25519_prod.pub     ← Public Key สำหรับ Production
├── id_ed25519_staging      ← Private Key สำหรับ Staging
├── id_ed25519_staging.pub  ← Public Key สำหรับ Staging
├── id_ed25519_github       ← Private Key สำหรับ GitHub
├── id_ed25519_github.pub   ← Public Key สำหรับ GitHub
└── config                  ← ไฟล์ตั้งค่าที่ผูกคีย์กับเซิร์ฟเวอร์
```

---

### B. ผูกแต่ละคีย์กับแต่ละเซิร์ฟเวอร์ผ่านไฟล์ `config`

เปิดไฟล์ `C:\Users\<ชื่อผู้ใช้>\.ssh\config` แล้วระบุ `IdentityFile` ให้ตรงกับคีย์ของแต่ละ Host:

```text
# Production Server (ใช้คีย์ prod)
Host prod
    HostName 103.224.86.15
    User deploy
    Port 22022
    IdentityFile ~/.ssh/id_ed25519_prod

# Staging Server (ใช้คีย์ staging)
Host staging
    HostName 192.168.1.50
    User deploy
    Port 22
    IdentityFile ~/.ssh/id_ed25519_staging

# GitHub (ใช้คีย์ github)
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github

# รูปแบบ Wildcard — เซิร์ฟเวอร์ทุกตัวใน subnet 10.0.0.x ใช้คีย์ชุดเดียวกัน
Host 10.0.0.*
    User admin
    IdentityFile ~/.ssh/id_ed25519_internal
    Port 22
```

จากนั้นล็อกอินได้ด้วยชื่อ alias ที่ตั้งไว้:
```bash
ssh prod       # เชื่อมต่อ Production ด้วยคีย์ prod อัตโนมัติ
ssh staging    # เชื่อมต่อ Staging ด้วยคีย์ staging อัตโนมัติ
```

---

### C. SSH Agent — จัดการคีย์หลายคู่โดยไม่ต้องพิมพ์ Passphrase ซ้ำทุกครั้ง

ถ้าตั้ง Passphrase ล็อกคีย์ไว้ (ซึ่งเป็นสิ่งที่ควรทำ) ระบบจะถาม Passphrase ทุกครั้งที่ใช้คีย์นั้น **SSH Agent** คือโปรแกรมที่รับจำ Passphrase ไว้ใน Memory แทน ทำให้ unlock คีย์ครั้งเดียวแล้วใช้ได้ตลอด session:

```bash
# 1. เปิด SSH Agent (บน PowerShell หรือ Linux/Mac Terminal)
# บน Linux/Mac:
eval "$(ssh-agent -s)"

# บน Windows PowerShell (ต้องรันในฐานะ Administrator ครั้งแรก):
Set-Service ssh-agent -StartupType Automatic
Start-Service ssh-agent

# 2. โหลดคีย์เข้า Agent (ระบบจะถาม Passphrase ครั้งเดียว)
ssh-add ~/.ssh/id_ed25519_prod
ssh-add ~/.ssh/id_ed25519_staging
ssh-add ~/.ssh/id_ed25519_github

# 3. ตรวจสอบคีย์ที่โหลดเข้า Agent แล้ว
ssh-add -l
# ผลลัพธ์ตัวอย่าง:
# 256 SHA256:abc123... prod-server (ED25519)
# 256 SHA256:def456... staging-server (ED25519)
# 256 SHA256:ghi789... your_email@example.com (ED25519)

# 4. ลบคีย์ออกจาก Agent (เมื่อต้องการปิดสิทธิ์ชั่วคราว)
ssh-add -d ~/.ssh/id_ed25519_prod   # ลบเฉพาะคีย์ prod
ssh-add -D                          # ลบคีย์ทั้งหมดออกจาก Agent
```

---

### D. การถอน Public Key ออกจากเซิร์ฟเวอร์ (Revoke Key)

เมื่อต้องการยกเลิกสิทธิ์ของคีย์ใดคีย์หนึ่ง (เช่น คีย์หาย, เปลี่ยนคอมพิวเตอร์, หรือเลิกให้ทีมใดเข้าถึง) ให้ล็อกอินเข้าเซิร์ฟเวอร์แล้วแก้ไขไฟล์ `authorized_keys`:

```bash
# เปิดไฟล์รายชื่อคีย์ที่ได้รับอนุญาต
nano ~/.ssh/authorized_keys
```

ไฟล์นี้จะมีคีย์เรียงเป็นบรรทัด แต่ละบรรทัดคือหนึ่ง Public Key เช่น:
```text
ssh-ed25519 AAAAC3Nza...abc prod-server
ssh-ed25519 AAAAC3Nza...def old-laptop
ssh-ed25519 AAAAC3Nza...ghi teammate-john
```

ลบบรรทัดของคีย์ที่ต้องการถอนออกไป บันทึกไฟล์ และสิทธิ์นั้นจะถูกยกเลิกทันทีโดยไม่ต้อง restart SSH

> [!WARNING]
> **ห้ามลบไฟล์ `authorized_keys` ทิ้งทั้งหมด** ถ้าคุณปิด Password Login ไว้ (ซึ่งแนะนำให้ทำใน Section 4) การลบไฟล์นี้จะทำให้ตัวเองล็อกอินเข้าเซิร์ฟเวอร์ไม่ได้เลยทันที

## 🔐 7. การตั้งค่า SSH Key ส่วนกลาง (Shared Key) และ Chroot Jail

ในบางองค์กรหรือทีมงาน อาจต้องการให้ผู้ใช้หลายคนล็อกอินเข้าเซิร์ฟเวอร์ด้วย **Public Key ชุดเดียวกัน** โดยไม่ต้องไปเพิ่มคีย์ใน `~/.ssh/authorized_keys` ของแต่ละ User ทีละคน แนวทางนี้ใช้ไฟล์คีย์กลาง (Shared Key File) ร่วมกับการล็อกสิทธิ์เข้าไดเรกทอรีด้วย Chroot Jail

---

### ขั้นตอนที่ 1: สร้างผู้ใช้งานที่จะใช้ Shared Key

สร้าง User ที่จะใช้เป็นบัญชีกลางสำหรับเชื่อมต่อ (เช่น `deploy`, `git`, `sftpuser`):

```bash
# สร้าง User ใหม่โดยไม่มี Home Directory และไม่มี Login Shell (เพื่อความปลอดภัย)
sudo useradd -M -s /usr/sbin/nologin sftpuser

# หรือถ้าต้องการให้ล็อกอิน SSH ได้แต่จำกัดสิทธิ์ด้วย Chroot ในขั้นตอนถัดไป
sudo useradd -m -s /bin/bash sftpuser
```

---

### ขั้นตอนที่ 2: เตรียมโครงสร้าง Chroot Jail Directory

**Chroot Jail** คือการบังคับให้ User มองเห็นและเข้าถึงได้เฉพาะโฟลเดอร์ที่กำหนดเท่านั้น แม้จะล็อกอินเข้ามาในระบบแล้วก็ตาม เปรียบเหมือนการขังอยู่ในกล่องทราย ออกไปข้างนอกไม่ได้

```bash
# สร้างโฟลเดอร์ที่จะใช้เป็น Chroot Jail (เจ้าของต้องเป็น root เท่านั้น)
sudo mkdir -p /srv/chroot/sftpuser

# กำหนดสิทธิ์ให้โฟลเดอร์ root ของ Chroot เป็นเจ้าของโดย root เสมอ
# (ข้อกำหนดบังคับของ OpenSSH — หากเจ้าของไม่ใช่ root ระบบจะปฏิเสธการ Chroot)
sudo chown root:root /srv/chroot/sftpuser
sudo chmod 755 /srv/chroot/sftpuser

# สร้างโฟลเดอร์ย่อยสำหรับให้ User เขียนไฟล์ได้
sudo mkdir -p /srv/chroot/sftpuser/uploads
sudo chown sftpuser:sftpuser /srv/chroot/sftpuser/uploads
sudo chmod 700 /srv/chroot/sftpuser/uploads
```

---

### ขั้นตอนที่ 3: ตั้งค่าไฟล์ SSH Key ส่วนกลาง (Shared Key)

แทนที่จะเก็บ `authorized_keys` ไว้ใน Home Directory ของแต่ละ User ให้นำ Public Key มาเก็บรวมไว้ที่ **ไฟล์กลาง** เพื่อให้ดูแลจัดการได้จากจุดเดียว:

```bash
# 1. สร้างโฟลเดอร์สำหรับเก็บคีย์กลาง
sudo mkdir -p /etc/ssh/shared_keys

# 2. สร้างไฟล์และนำ Public Key วางลงไป
sudo nano /etc/ssh/shared_keys/sftpuser_key
# (วาง Public Key ของผู้ที่ได้รับอนุญาต แต่ละบรรทัดคือหนึ่งคีย์)

# 3. ตั้งสิทธิ์ให้ปลอดภัย (root เป็นเจ้าของ, คนอื่นอ่านได้อย่างเดียว)
sudo chown root:root /etc/ssh/shared_keys/sftpuser_key
sudo chmod 644 /etc/ssh/shared_keys/sftpuser_key
```

---

### ขั้นตอนที่ 4: คอนฟิก SSH Daemon (`sshd_config`)

เปิดไฟล์ตั้งค่าหลักของ SSH เพื่อบอกให้ระบบทำ Chroot Jail และชี้ไปอ่านคีย์จากไฟล์กลางที่สร้างไว้:

```bash
sudo nano /etc/ssh/sshd_config
```

เลื่อนลงไปที่ล่างสุดของไฟล์ แล้วเพิ่มโค้ดชุดนี้ต่อท้ายเข้าไป:

```text
# ===== Shared Key + Chroot Jail สำหรับ sftpuser =====
Match User sftpuser
    # ชี้ไปอ่าน Public Key จากไฟล์กลางแทนที่จะอ่านจาก ~/.ssh/authorized_keys
    AuthorizedKeysFile /etc/ssh/shared_keys/sftpuser_key

    # ล็อกให้ User เห็นเฉพาะโฟลเดอร์ที่กำหนด (Chroot Jail)
    ChrootDirectory /srv/chroot/sftpuser

    # จำกัดให้ใช้ได้เฉพาะ SFTP เท่านั้น (ไม่สามารถเปิด Shell หรือ Port Forwarding ได้)
    ForceCommand internal-sftp
    AllowTcpForwarding no
    X11Forwarding no
```

> [!TIP]
> **`Match User`** คือ Directive พิเศษของ SSH ที่ใช้สร้างกฎเฉพาะสำหรับ User ที่ระบุ กฎที่เขียนภายใต้บล็อกนี้จะ Override การตั้งค่าทั่วไปด้านบนเฉพาะ User คนนั้นเท่านั้น สามารถใช้กับกลุ่มได้ด้วย `Match Group sftp-users`

### ขั้นตอนที่ 5: รีสตาร์ต SSH และทดสอบ

```bash
# ตรวจสอบ syntax ของ sshd_config ก่อน restart (สำคัญมาก!)
sudo sshd -t
# ถ้าไม่มี output ใดๆ = ไฟล์ถูกต้อง พร้อม restart ได้

# รีสตาร์ต SSH Daemon
sudo systemctl restart sshd
```

> [!WARNING]
> **ข้อกำหนดบังคับของ Chroot Jail ใน OpenSSH:** โฟลเดอร์ที่ระบุใน `ChrootDirectory` และ **ทุกโฟลเดอร์แม่** ในเส้นทางนั้น **ต้องมีเจ้าของเป็น `root` และห้ามให้ Group/Others มีสิทธิ์ Write** มิฉะนั้น SSH จะปฏิเสธการเชื่อมต่อทันทีพร้อม Error `bad ownership or modes for chroot directory`
